---
title: "Secrets scanning"
description: "Detect leaked API keys, tokens, and credentials in your codebase with Semgrep Secrets."
---

Semgrep Secrets finds sensitive credentials committed to source code — API keys, authentication tokens, private keys, connection strings, and more. It combines semantic code analysis, high-entropy string detection, and optional **live validation** to confirm whether discovered credentials are active.

<Note>
Full Secrets scanning — including the managed ruleset and live credential validation — requires logging in to the Semgrep AppSec Platform. Run `semgrep login` first, or use `semgrep ci` in a connected CI environment.
</Note>

## How Semgrep Secrets works

<Steps>
  <Step title="Semantic pattern matching">
    Semgrep applies rules that understand code structure — not just regex over raw text. A rule can match an assignment like `api_key = "..."` regardless of surrounding whitespace or string quoting style.
  </Step>
  <Step title="Entropy analysis">
    High-entropy strings (sequences that look random) are flagged as potential secrets even without a named variable context. This catches tokens embedded in URLs, config objects, or comments.
  </Step>
  <Step title="Validation">
    For supported credential types, Semgrep makes an outbound HTTP request to the relevant API to check whether the credential is valid (i.e., actively accepted). Only validated credentials are promoted to high-severity findings.
  </Step>
</Steps>

## Enabling Secrets scanning

### In CI

Add the `--secrets` flag to your `semgrep ci` invocation:

```bash
semgrep ci --secrets
```

Or configure it in your Semgrep AppSec Platform policy to always enable Secrets for all scans.

### Locally

```bash
semgrep --config p/secrets .
```

<Tip>
The community `p/secrets` ruleset covers common credential patterns but does not include live validation. For validation, use `semgrep ci --secrets` with a logged-in session.
</Tip>

## Validation

When validation is enabled, Semgrep sends the discovered credential to the issuing service's API and checks whether it is accepted. The result appears as the `validation_state` in the finding:

| Validation state | Meaning |
|---|---|
| `confirmed_valid` | The credential was accepted by the target API — it is live and should be rotated immediately. |
| `confirmed_invalid` | The credential was rejected — it may be expired, already rotated, or a false positive. |
| `validation_error` | Validation could not be completed (e.g., network timeout or API rate limit). |
| `no_validator` | No validation rule exists for this credential type. |

In JSON output:

```json
{
  "check_id": "semgrep.secrets.github-pat",
  "path": "config/deploy.sh",
  "extra": {
    "message": "GitHub Personal Access Token found.",
    "severity": "ERROR",
    "validation_state": "confirmed_valid",
    "metadata": {
      "secret_type": "GitHub Personal Access Token"
    }
  }
}
```

## `--secrets-timeout`

Validation requests have a default timeout of **30 seconds**. You can override this:

```bash
semgrep ci --secrets --secrets-timeout 60
```

Set to `0` to disable the timeout entirely (not recommended in CI).

## Common secret types detected

| Secret type | Example pattern |
|---|---|
| AWS access keys | `AKIA[0-9A-Z]{16}` |
| GitHub Personal Access Tokens | `ghp_[A-Za-z0-9]{36}` |
| GitHub App tokens | `ghs_[A-Za-z0-9]{36}` |
| Slack tokens | `xox[baprs]-[0-9A-Za-z-]{10,48}` |
| Stripe API keys | `sk_live_[0-9a-zA-Z]{24}` |
| Twilio auth tokens | 32-character hex strings paired with `ACCOUNT SID` |
| Private SSH/TLS keys | `-----BEGIN (RSA|EC|OPENSSH) PRIVATE KEY-----` |
| Generic connection strings | `postgres://user:password@host/db` |
| Generic high-entropy strings | Entropy-based detection without a specific pattern |

## Suppressing false positives

If Semgrep Secrets flags a value that is intentionally not sensitive (e.g., a test credential, a placeholder, or an example in documentation), suppress it with an inline comment:

```python
API_KEY = "AKIAIOSFODNN7EXAMPLE"  # nosemgrep: semgrep.secrets.aws-access-key
```

Or suppress all rules on a line:

```python
AWS_SECRET = "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"  # nosemgrep
```

For broader suppression across a file section, add a block-level `nosemgrep` comment:

```python
# nosemgrep
EXAMPLE_CREDENTIALS = {
    "access_key": "AKIAIOSFODNN7EXAMPLE",
    "secret_key": "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY",
}
# nosemgrep: end
```

<Warning>
Suppress findings only when you are confident the value is not a real credential. Suppressing valid secrets leaves your system exposed.
</Warning>

## Secrets in CI

When running with `semgrep ci --secrets`, confirmed-valid secrets can be configured to block pull requests in the Semgrep AppSec Platform. Example GitHub Actions workflow:

```yaml
# .github/workflows/semgrep-secrets.yml
name: Semgrep Secrets
on:
  push:
    branches: [main]
  pull_request: {}
jobs:
  secrets-scan:
    runs-on: ubuntu-latest
    container:
      image: semgrep/semgrep
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - run: semgrep ci --secrets
        env:
          SEMGREP_APP_TOKEN: ${{ secrets.SEMGREP_APP_TOKEN }}
```
